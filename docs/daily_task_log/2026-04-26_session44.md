## 교훈 / 학습

### per-row 파싱 fault tolerance — `feedback_per_row_parse_tolerance.md` 신규

리스트 fetch 의 `.map((json) => DTO.fromJson(json)).toList()` 는 한 row 만 깨져도 전체가 폭파되어 `Failure.unknown` 으로 떨어진다. Repository 가 캐시 fallback 하면 캐시 일부만 보여 사용자 데이터 가시성 회귀가 발생한다. `LinkRemoteDataSource.parseRows` 가 표준 헬퍼 — per-row try/catch + `appLogger.w` 로 실패 row 만 skip + 로깅. 신규 list fetch 추가 시 동일 패턴 답습할 것.

### 진단 우선 + 스코프 분리

사용자가 "태그 있는 링크만 사라짐" 보고 → Stage 2 (문자열 변경) 와 인과 무관함을 즉시 검증한 뒤 별도 핫픽스로 분리. Stage 2 branch 는 안전하게 보존하고 main 으로 돌아가서 핫픽스 작업 → main rebase → resume 흐름이 깔끔하게 동작.

### `flutter run` 백그라운드 모니터링으로 자체 진단

`flutter run` 을 Monitor tool 로 백그라운드 + grep 필터 (`failed to parse|LinkRemoteDataSource|TypeError`) 적용해서 Claude 가 자체적으로 실기기 진단 가능. 사용자에게 `adb logcat` 떠 달라고 부탁할 필요 없음 — 빌드 + 설치 + 실행 + 로그 캡처를 한 번에 처리.
