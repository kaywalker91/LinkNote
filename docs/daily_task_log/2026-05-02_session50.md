## 미션

Phase 4 잔여 후보 Dark mode forest 팔레트 튜닝. 기존 `ThemeData.dark()` 는 slate-blue 1차 iteration 으로 forest 정체성과 어긋나 있었고, 더 큰 문제는 **Ln 위젯이 `AppColors.X` 를 직접 하드코딩** → ThemeData 만 바꿔서는 색이 안 변하는 구조적 결함. AppPalette `ThemeExtension` 도입으로 brightness-aware 토큰 인프라 + Ln 위젯 마이그레이션 + dark forest 팔레트 + 회귀 테스트 단일 PR.

## 디자인 결정 노트

**Why ThemeExtension 대신 단순 `AppColors.*Dark` 추가가 부적합**:
- Ln 위젯이 `AppColors.ink` / `AppColors.bg` 직접 참조 → ThemeData 만 다크 전환해도 위젯 색은 light 고정.
- `Theme.of(c).colorScheme.X` 로 모두 옮기는 것은 디자인 토큰 22개를 Material 6개 슬롯에 억지로 맞춰야 함 → 의미 손실.
- ThemeExtension 은 정확히 이런 도메인 토큰을 brightness 별 변형하는 canonical 패턴(Flutter 공식 권장).

**Why structural 회귀 테스트 (vs 시각 golden)**:
- `golden_toolkit` 미설치 → 폰트 / 픽셀 라스터화가 OS / 머신 별 비결정적.
- CI 가 `flutter test --coverage` 를 Ubuntu 에서 실행 → macOS 로컬 generated golden 매칭 실패 위험.
- 구조적 색·렌더 어설션은 동등한 회귀 보호 + cross-platform 결정적.
- 시각 golden 도입은 `golden_toolkit` / 폰트 정규화 후 별도 세션.

**Light 회귀 zero 보장 검증 경로**:
- `test/app/theme/app_palette_test.dart` 의 첫 케이스 — `AppPalette.light()` 의 16 토큰 모두 `AppColors.X` 와 동등 비교.
- `test/shared/widgets/ln/ln_link_card_test.dart` 등 기존 widget 테스트(`MaterialApp(home: ...)` 패턴)는 fallback 으로 light palette 받아 byte-identical 동작.
